# NearbyLove × CommunitySense 项目说明

## 1. 项目概述

### 1.1 项目定位

NearbyLove × CommunitySense 是面向社区数智化管理的协同平台原型。项目不把社区治理理解为单纯的“发现问题—派单—结案”，而是围绕共同生活中的具体空间与事务，建立居民、物业、居委会和政府之间可理解、可协作、可验证的行动闭环。

项目由两部分组成：

- **NearbyLove**：面向居民、物业、居委会的三端协同体验。居民可以了解附近服务、查看官方通知、提出或参与附近行动、使用社区信箱；物业围绕设施运行地图和履约任务开展现场工作；居委会围绕区域关怀地图、居民诉求摘要和行动协调组织公共服务。
- **CommunitySense**：面向镇政府的治理大屏。以社区为统计粒度呈现人口、关系连接和社区行动的区域总览与分社区变化，用于观察治理重点与改善趋势。

### 1.2 核心问题

高流动性社区中，实际居住者、公共空间状态和居民诉求往往分散在不同主体与线下流程中：新居民不知道向谁咨询，沉默者的公共需要难以被看见，物业难以从零散反馈中判断设施维护重点，居委会难以持续组织可验证的公共行动，政府也缺少跨社区的结构化观察视角。

项目以“人—场—事”形成回应：

| 层次 | 产品机制 | 回答的问题 |
|---|---|---|
| 人 | 虚拟户口本 | 谁在这里实际生活、谁可获得相应服务？ |
| 场 | 公共空间 AI 感知 | 这片共同生活的空间正在发生什么？ |
| 事 | 附近行动 | 居民、物业和居委会如何共同改善它？ |

### 1.3 核心创新与边界

1. **从事务响应走向附近行动。** 产品将居民反馈、物业履约和居委会协调围绕同一具体议题组织，而不是把社交、工单、通知和治理数据割裂为彼此独立的功能。
2. **公共空间匿名感知。** AI 硬件定位为公共空间的非人格化感知层：只输出可供人工核实的场景线索，不识别人脸、身份、轨迹或个体行为，不自动处罚。
3. **AI 社区管家是辅助者而非决策者。** AI 可整理通知反馈、归纳居民诉求、引导下一步操作和生成待核实线索；无法回答时只引导居民写入可追溯的社区信箱，不替代人工正式回复或自动转人工。
4. **安全责任回归官方服务主体。** 自愿安全服务必须由本人授权，紧急责任由依法配置的社区服务主体承担，不转嫁给邻居或平台算法。

## 2. 产品与演示内容

### 2.1 三端协同体验

`web/nearbylove-demo.html` 提供可在浏览器中直接体验的居民、物业、居委会三端网页原型。

| 端 | 页面主线 | 代表功能 |
|---|---|---|
| 居民端 | 看懂附近—提出议题—参与行动—查看结果 | 500 米指南、官方通知、社区广场、附近行动、AI 社区管家、社区信箱、虚拟户口本与服务入口 |
| 物业端 | 设施运行—现场核实—任务履约—结果反馈 | 设施运行地图、巡检/报修任务、AI 场景信、通知发布与居民反馈纪要 |
| 居委会端 | 区域关怀—议题协调—行动组织—效果验证 | 区域关怀地图、行动协调、社区信箱、官方通知、AI 工作摘要 |

### 2.2 政府治理大屏

`web/communitysense-dashboard.html` 提供政府端网页大屏原型。大屏以“区域总览—社区分块—社区详情”的结构，聚焦三个问题：

- 区域中有哪些人口和居住关系变化？
- 不同社区的关系连接与服务可达性如何？
- 附近行动、物业履约和跨部门协同是否产生可观察的改善？

### 2.3 后端与硬件技术验证

Flask 后端提供健康检查、传感器读数接收、区域汇总、事件、AI 线索分诊和自愿关怀任务等 API。`scripts/seed_demo_areas.py` 与 `scripts/simulate_sensor.py` 可写入模拟公共空间读数；`hardware/` 中保留 OpenMV 串口/图片采集验证代码。

本仓库的在线网页演示使用 Mock 数据，评委无需配置账号、硬件、密钥或数据库即可体验。后端和硬件代码用于说明完整技术链路，并可按 README 在本地运行。

## 3. 技术栈

| 层次 | 技术与版本 | 用途 |
|---|---|---|
| 网页原型 | HTML5、CSS3、原生 JavaScript | 三端协同网页与在线演示入口，无构建步骤 |
| 数据可视化 | ECharts 5.5.0、Leaflet 1.9.4 | 政府治理大屏中的图表与地图；通过公开 CDN 在浏览器加载 |
| 后端 API | Python 3.11+、Flask 3.1.3、Flask-CORS 6.0.5 | REST API、跨域开发支持 |
| 数据层 | Flask-SQLAlchemy 3.1.1、SQLAlchemy 2.0.51、SQLite | 原型阶段的本地持久化与数据模型 |
| AI 接口 | OpenAI Python SDK 2.45.0、python-dotenv 1.2.1 | 可选的兼容 OpenAI API 技术验证；默认 Mock 模式不需要密钥 |
| 硬件/模拟 | pyserial 3.5、Pillow、requests | OpenMV 串口读取、图像亮度处理、传感器读数上报与模拟 |
| 部署 | GitHub Pages、GitHub Actions | 静态网页入口自动部署；后端不在 Pages 上运行 |

完整 Python 依赖和锁定版本见仓库根目录 `requirements.txt`。

## 5. 未来规划

### 5.1 不同社区场景的迁移与定制化服务

1. 将当前原型沉淀为可配置的社区模板，根据老旧小区、商品房社区、流动人口聚集区、城乡接合部等不同场景，调整服务目录、空间点位、行动议题、物业协作流程与居委会关怀重点。
2. 以社区实际人口结构、空间形态、物业服务能力和既有数字化基础为输入，配置居民端 500 米指南、物业设施地图、居委会区域关怀地图和政府端观察指标，避免“一套功能覆盖所有社区”。
3. 在每个试点保留统一的隐私底线与行动闭环，同时允许居民、物业和居委会共同定义本地高频问题及可接受的改善方案。

### 5.2 AI 硬件摄像头算法模型提升

1. 在不识别个人身份、不追踪个体轨迹的前提下，提升公共空间场景识别能力，优先覆盖消防/无障碍通道受阻、公共设施失效、服务点拥堵和公共区域异常堆放等低侵入场景。
2. 优化边缘端脱敏、场景结构化、置信度评估和复发检测算法，使硬件输出“待人工核实的行动线索”，而不是未经解释的告警或对人的判断。
3. 建立“硬件线索—物业核实—居委会协调—居民反馈—改善验证”的样本反馈机制，用整改是否减少同类问题、是否改善公共空间可用性来评估算法价值，而不是以告警数量作为目标。
4. 在真实试点前完善点位告知、数据最小化、短期留存、访问审计和人工复核机制，确保算法升级始终服务于公共安全、老年人/儿童友好和社区共同生活，而非强化监控。
